// 01 — PROJECT OVERVIEW

What This System Does

This embedded project implements a 4-degree-of-freedom robotic arm controlled entirely by an Arduino Uno microcontroller. Four MG996R high-torque servos drive the base rotation, shoulder, elbow, and wrist joints — each independently mapped through analog potentiometer inputs.

The defining capability is position memory: the operator manually guides the arm to desired poses by adjusting the potentiometers, presses a record button to capture the joint-angle snapshot into SRAM, then triggers playback to replay the full recorded trajectory. The system interpolates smoothly between waypoints rather than snapping joint-to-joint, producing fluid, repeatable motion sequences.

All computation — angle mapping, array management, interpolation math, PWM generation — runs bare-metal on the Uno's ATmega328P at 16 MHz. No RTOS, no libraries beyond the Arduino Servo framework. Pure embedded control.

⚙ Teach-by-demonstration paradigm: human sets the path, the machine repeats it indefinitely with sub-degree precision.

// 02 — SYSTEM FEATURES

Core Capabilities

01

Manual Potentiometer Control

Four 10kΩ potentiometers provide continuous analog voltage signals to the Arduino's ADC channels. Each 10-bit reading (0–1023) maps linearly to a servo-specific angular range, enforcing safe mechanical limits per joint. Control is instantaneous with zero latency.

ADC · 10-bit · Linear Mapping
02

Position Recording

A dedicated record button triggers a snapshot of all four joint angles into a sequential SRAM array. Up to 50 waypoints can be captured per session. Each record event is debounced in firmware to eliminate false triggers, and the current index is tracked for overwrite protection.

SRAM Array · Debounce · Snapshot
03

Smooth Trajectory Playback

During playback, the system iterates through stored waypoints and generates intermediate angle values via linear interpolation between consecutive positions. Servos update at 50 Hz, producing fluid, continuous motion rather than discrete jumps — critical for real-world manipulation tasks.

50 Hz PWM · Lerp · Trajectory
04

Adjustable Playback Speed

A dedicated analog input or multi-mode button cycles through playback speed presets — slow (high step count), normal, and fast. Speed adjustment alters the interpolation step density between waypoints, enabling both precise inspection of motion and rapid cycle execution.

Step Density · Multi-mode · Cycle Time
05

Safety Angle Constraints

Each servo joint is individually constrained to a validated angular envelope defined in firmware constants. Regardless of raw potentiometer input, the mapped output is clamped before being written to the servo. This prevents over-rotation, physical collision between arm links, and mechanical damage to gears under load.

Clamp · Per-Joint Limits · Collision Avoidance
06

Real-Time Loop Architecture

The firmware runs a deterministic 20 ms control loop. On each iteration, ADC reads, angle computation, servo writes, and button polling all complete within the loop budget. This guarantees consistent servo refresh rates and eliminates jerky motion artifacts caused by variable execution timing.

20 ms Loop · Deterministic · ATmega328P

// 04 — WORKING PRINCIPLE

System Architecture

LAYER 01

Input Layer — Potentiometer Sensing

Each of the four 10kΩ potentiometers forms a voltage divider fed into an analog pin. The ATmega328P's 10-bit successive-approximation ADC samples each channel sequentially at the start of every control loop iteration. Readings are averaged over 4 samples per channel to reduce quantization noise before being forwarded to the mapping stage. 

// ADC read with oversampling
int raw = 0;
for(int i = 0; i < 4; i++) raw += analogRead(POT_PIN);
int angle = map(raw / 4, 0, 1023, JOINT_MIN, JOINT_MAX);
▼
LAYER 02

Processing Layer — Angle Mapping & Storage

Oversampled ADC values are passed through the Arduino map() function to translate the 0–1023 domain into per-joint angular limits stored as firmware constants. A constrain() call enforces hard safety bounds before the angle is used. When the record button is pressed, all four current angles are written into a 2D integer array (positions[50][4]), and the sequence index is incremented. 

// Record snapshot
positions[recordIndex][0] = angle[BASE];
positions[recordIndex][1] = angle[SHOULDER];
positions[recordIndex][2] = angle[ELBOW];
positions[recordIndex][3] = angle[WRIST];
recordIndex++;
▼
LAYER 03

Output Layer — Servo PWM Control

The Arduino Servo library generates 50 Hz PWM signals with pulse widths from 544 µs to 2400 µs, mapping 0°–180°. Each servo is driven on a dedicated digital pin. During manual mode, computed angles are written directly to servos each loop cycle. During playback, interpolated intermediate angles are written instead, advancing the arm smoothly through the recorded trajectory. 

// Write to all joints
servo[BASE].write(currentAngle[BASE]);
servo[SHOULDER].write(currentAngle[SHOULDER]);
servo[ELBOW].write(currentAngle[ELBOW]);
servo[WRIST].write(currentAngle[WRIST]);
▼
LAYER 04

Playback — Linear Interpolation Engine

Between each pair of consecutive waypoints, the playback engine computes N intermediate frames (where N is determined by the speed setting). For each frame t ∈ [0, N], the angle for joint j is: angle = start[j] + (end[j] − start[j]) × (t / N). Each frame is written and held for a fixed delay, producing smooth, continuous servo motion. The loop concludes by optionally cycling back to the first waypoint. 

// Interpolate between waypoints
for(int t = 0; t <= STEPS; t++) {
  float alpha = (float)t / STEPS;
  for(int j = 0; j < 4; j++)
    servo[j].write(start[j] + (end[j]-start[j]) * alpha);
  delay(FRAME_DELAY);
}

Engineering Deep Dive

PWM Servo Control

Standard RC servo protocol uses a 50 Hz (20 ms period) PWM signal. Pulse width between 1 ms and 2 ms commands 0°–180°. The ATmega328P's Timer1 16-bit hardware timer generates these pulses in hardware, freeing the CPU for computation. The Arduino Servo library abstracts this, but write() ultimately writes to the OCR1A/B compare registers directly.

Timer1OCR1A/B50 Hz1–2 ms pulse

Array-Based Memory Storage

Joint positions are stored in a statically allocated 2D integer array: int positions[MAX_POS][NUM_JOINTS]. This occupies 400 bytes of SRAM (50 × 4 × 2 bytes) on the ATmega328P's 2kB SRAM — tight but feasible. Index management, boundary checks, and reset logic run in O(1) time with no dynamic allocation, ensuring deterministic execution.

Static SRAMO(1) access2D int array400B footprint
∿

Motion Interpolation Algorithm

Linear interpolation (lerp) generates a smooth velocity profile between discrete waypoints. While not as smooth as cubic spline or trapezoidal motion profiles, lerp is computationally trivial — one multiply and one add per joint per frame — keeping loop latency under budget. Future upgrades may implement minimum-jerk polynomial trajectories for smoother acceleration and deceleration.

LerpWaypointsVelocity ProfileO(N·J)
⊞

Mechanical Safety Constraints

Firmware-level constraints prevent commanding any joint beyond its physical travel. Per-joint min/max constants are derived from mechanical measurement of the arm's safe range, accounting for link clearances and cable management. The constrain() function ensures voltage noise or software bugs can never produce an out-of-range write, preventing gear stripping and structural collision.

// 06 — DEVELOPMENT ROADMAP

Future Improvements

PLANNED
💾

EEPROM Persistent Storage

Migrate position arrays from volatile SRAM to the ATmega328P's 1kB EEPROM. Sequences survive power cycles and can be recalled without re-teaching — essential for production-use automation fixtures.

PLANNED
📺

LCD Status Display

Add an I²C 16×2 LCD to display current mode (MANUAL / RECORD / PLAY), active joint angles, recorded position count, and playback speed — eliminating guesswork during operation.

RESEARCHING
📡

Bluetooth Wireless Control

Replace physical potentiometers with an HC-05 Bluetooth module and a smartphone slider interface. Commands transmitted as serial packets over UART, enabling wireless teach-and-playback from any Android device.

RESEARCHING
🔢

Inverse Kinematics Engine

Implement a geometric or Jacobian-based IK solver to accept Cartesian coordinates (X, Y, Z) as input and compute the required joint angles automatically — shifting from joint-space to task-space control.

CONCEPT
🌐

Web Dashboard Integration

Pair with an ESP32 Wi-Fi module to expose a REST API and browser-based dashboard for real-time joint angle visualization, sequence upload/download, and remote playback triggering over a local network.

CONCEPT
⚡

Minimum-Jerk Trajectory Planning

Replace linear interpolation with 5th-order polynomial (minimum-jerk) motion profiles, producing smooth acceleration/deceleration envelopes. This reduces mechanical stress on gears and improves precision at trajectory endpoints.
